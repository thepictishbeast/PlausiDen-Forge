# PlausiDen-Forge — design

> **Audience:** new contributor, AI agent, or auditor. Read this
> before opening source. Read [DEPRECATION.md](./DEPRECATION.md)
> first to know which paths are live vs migration-only.

## What forge is

A **typed, audited build pipeline** for static + hybrid sites
authored against the Loom design system. One invocation:

```
cargo run -p forge-cli --release
```

…runs every phase against the project's `cms/`, `static/`, and
`backends.toml`, accumulates `Finding`s, and gates the exit code
on whether any strict findings landed.

Forge is also the runtime audit driver — it shells out to the
sibling [PlausiDen-Crawler](https://github.com/thepictishbeast/PlausiDen-Crawler)
in its final phase, so one report covers build + render + audit.

## Crate map

| Crate | Role | When you touch it |
|---|---|---|
| `forge-core` | `Phase` trait, `Finding`, `Severity`, `BuildCtx`, `BuildReport`, `BuildError`. Pure types. | Almost never. Adding a finding kind is a doctrine change. |
| `forge-phases` | One module per phase (22+). Each `impl Phase`. | Adding a phase, fixing an existing check. |
| `forge-cli` | Binary. Parses argv, builds the phase pipeline, runs it, emits the report. | Re-ordering phases, adding CLI flags. |
| `forge-serve` | Live-reload dev server (queued). | Not yet. |
| `forge-replay` | Replay a prior build report. | Not yet. |
| `server-stub` | Per-`backends.toml` entry handler scaffolds. Generated by `loom backend-stub`. | Don't hand-edit; regenerate. |

## Phase doctrine

Every phase implements:

```rust
pub trait Phase: Send + Sync {
    fn name(&self) -> &'static str;
    fn run(&self, ctx: &BuildCtx) -> Result<Vec<Finding>, BuildError>;
}
```

Constraints:

1. **Pure-input → pure-output.** A phase MUST NOT mutate the
   `BuildCtx`. The runner is free to parallelize phases in the
   future; mutable state breaks that.
2. **No `unwrap` / `expect` in lib code.** Lint configured at the
   crate level. Errors flow through `BuildError`.
3. **Findings are typed by `Severity`.** `Strict` blocks ship
   regardless of mode; `Warn` is suppressible in PoC mode.
4. **ADT findings, not stringly-typed.** New phases SHOULD wrap
   their internal results in an `enum FooFinding { … }` and have
   an `as_finding()` renderer. See `theme_consistency.rs` and
   `crawl.rs` for the pattern.
5. **Value Objects at every input boundary.** A `--loom-color-X`
   token name is `TokenName(String)` with constructor validation,
   not `&str`. Wildcards, traversal, shell metachars die at the
   constructor.
6. **Property-based tests** for any phase that ingests arbitrary
   bytes. See `theme_consistency::proptests` for the template.
7. **`// SECURITY:`, `// REGRESSION-GUARD:`, `// BUG ASSUMPTION:`,
   `// SAFETY:` annotations** inline. Grep-discoverable for
   audits.

## Pipeline order (forge-cli/src/main.rs)

1. `loom_sync` — verify `static/loom-skin.css` matches Loom upstream
2. `self_check` — meta-audit of forge itself
3. `theme_consistency` — `var(--loom-color-X)` resolves to a base
4. `tokens` — magic-number sweep
5. `html_semantic` — landmark + heading order
6. `csp` / `csp_devmode` — strict CSP headers
7. `external_assets` — third-party leak detector
8. `a11y_landmarks` — ARIA structure
9. `id_strategy` — id collision + uniqueness
10. `seo` — `<title>` / meta / canonical
11. `perf_budget` — payload ceilings
12. `asset_optimization` — gzip + brotli pre-compress
13. `sri` — Subresource Integrity hashes
14. `phantom_button` — buttons without `data-backend`
15. `backend_coverage` — backends.toml ↔ HTML ref consistency
16. `unbuilt_route` — link to nonexistent page
17. `label_consistency` — same `(href, data-backend)` → one label
18. `link_check` — every internal anchor resolves
19. `motion` — animation durations are tokenized
20. `contrast` — WCAG AA on every fg/bg
21. `crawl` — runtime audit via PlausiDen-Crawler

Build-infra issues surface earlier; runtime-only regressions get
their own rung so the operator can tell them apart.

## Severity rules

| Mode | What blocks ship |
|---|---|
| `Poc` | Strict findings only. Warns surface in the overlay but pass build. |
| `Production` | Strict + Warn both block. Default for `forge --mode production`. |
| `Static` | Implies static-only output. Phase set unchanged. |
| `Hybrid` | Static + dynamic. Future. |
| `Dynamic` | Pure dynamic backend. Future. |

## How to add a new phase

1. Create `crates/forge-phases/src/<name>.rs`.
2. Define an `enum <Name>Finding { … }` ADT for results.
3. Define a `<Name>Phase` ZST + `impl Phase`.
4. Use Value Objects for any string input.
5. Annotate with `// SECURITY:`, `// REGRESSION-GUARD:`, etc.
6. Write deterministic tests AND ≥1 proptest target.
7. Register `pub mod <name>;` in `crates/forge-phases/src/lib.rs`.
8. Wire `Box::new(<Name>Phase)` into the phase list in
   `crates/forge-cli/src/main.rs`.
9. `cargo test -p forge-phases` must pass.
10. `cargo run -p forge-cli` against the live PoC produces the
    expected findings.

## How to fix a finding

1. Read the phase's `as_finding()` to find the suggestion text.
2. Edit the source (CMS file, skin.css, `backends.toml`, etc.).
3. Re-run forge.
4. Repeat until 0 strict findings.

`Warn` findings are non-blocking but should drift to 0 over time.

## How forge integrates with Loom

Loom owns the typed primitives (`CmsPage`, `CmsSection`, themes).
Forge is the orchestrator that uses Loom subcommands as steps:

- `loom validate` (CMS schema check) → `phase_validate_cms`
- `loom audit-bridge` (skin.css ↔ CmsSection) → `phase_audit_bridge`
- `loom theme validate` → `phase_theme_consistency` (Rust port)

When Loom changes its CLI surface, the corresponding Rust phase
in forge-phases is the integration point. There is no shell-out
when both sides are Rust — phases call directly into Loom crates
when possible.

## How forge integrates with Crawler

`phase_crawl` (Rust, T52) shells out to
`npm run audit -- --journey <…>`, parses the latest run's
`runs/<ts>/diff.json` via typed serde, and surfaces strict axis
regressions as forge strict findings.

The crawler dir is resolved via `CRAWLER_DIR` env var first,
falling back to conventional sibling paths. Missing crawler
degrades to `Warn` — devs without the sibling repo aren't
blocked from building.

## How forge integrates with Hetzner / deployment

Not yet. T47 (`loom deploy hetzner`) will:

1. Verify `forge build` exits 0.
2. `rsync static/` to `/var/www/<tenant>/publish-<sha>/` over SSH.
3. Atomic symlink swap: `publish/` → `publish-<sha>/`.
4. Reload nginx.
5. Curl every declared page URL, assert 200 + SHA-pinned overlay.
6. Retain previous symlink target for one swap; `loom deploy
   rollback` flips back.

## Reports

Every forge run writes:

- `reports/latest.json` — typed `BuildReport` + every finding.
- `reports/build-<ISO8601>.json` — same, time-stamped.
- `static/forge-findings.js` — `window.__FORGE_FINDINGS__` —
  consumed by the in-page overlay.

T26 (queued) will SHA-256-chain the time-stamped reports so any
mutation or omission is detectable. The chain root + length will
land in `reports/latest.json`.

## Testing strategy

- **Unit tests** — colocated in each phase module. Cover the
  pure parsers + classification logic.
- **Property-based tests** — `proptest` workspace dep. Mandatory
  for any phase that ingests arbitrary bytes (parsers, JSON
  consumers). Proves the absence of panics across a search of
  1024+ cases per property.
- **Integration tests** — TODO. T44 sketched fast-rebuild infra
  that should also gain end-to-end smoke against a fixture site.
- **Mutation testing** — TODO. AVP-2 Tier 6 mandates
  `cargo mutants` < 5% survival on critical paths.

## Security model

Forge runs LOCAL on the developer's machine OR INSIDE a per-tenant
sandbox (T46). It does NOT run as a network service in v1. Threats
considered:

- **Path traversal in CMS/backends/skin paths** — every input
  flows through a Value Object constructor. `BackendKey`,
  `TokenName`, `AxisName` reject anything outside their schema.
- **Shell injection in phase commands** — every shelled-out
  invocation uses `Command::new(...).arg(...)` with arg-vec.
  No `bash -c` strings constructed from user data.
- **Untrusted JSON parse panics** — typed `serde::Deserialize`
  with `#[serde(default)]` on every field. Proptest target proves
  no panic on arbitrary input.
- **Output integrity** — T26 will Merkle-chain build reports.
- **Trust boundary** — forge runs as the user; it produces
  artefacts the user would have produced anyway. Privileged
  operations (deploy, etc.) live in separate binaries with
  capability tokens (T25).

## Glossary

- **AVP-2** — Adversarial Validation Protocol, version 2. The
  user's project-wide quality doctrine.
- **Supersociety stack** — the AVP-2 layered defence stack:
  formal verification → memory-safe lang → property + mutation
  testing → fuzzing → static analysis → dynamic analysis →
  supply-chain verification → runtime defence in depth →
  cryptographic integrity.
- **Loom** — the typed design system + primitives layer.
- **Crawler** — the headless-browser audit driver.
- **Phase** — one step of the forge pipeline (e.g.
  `theme_consistency`).
- **Finding** — a single result emitted by a phase, with phase
  name, path, message, and severity.
- **Strict finding** — a finding that always blocks ship,
  regardless of mode.
